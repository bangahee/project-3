# Mini NPU Simulator

## 1. 프로젝트 개요

이 프로젝트는 입력된 패턴이 어떤 필터와 더 유사한지를 판단하는 Python 콘솔 기반 프로그램이다.  
패턴과 필터를 2차원 배열 형태로 표현하고, MAC(Multiply-Accumulate) 연산을 통해 유사도를 계산한다.

사용자는 직접 3×3 필터와 패턴을 입력하거나, JSON 파일에 저장된 데이터를 이용하여 여러 패턴을 자동으로 분석할 수 있다.

## 3. 구현 요약

### 3.1 MAC 연산

MAC(Multiply-Accumulate) 연산은 입력 패턴과 필터의 같은 위치 값을 곱한 후 모두 더하여 점수를 계산하는 방식이다.

```python
total += pattern[i][j] * filter[i][j]
```

코드 구조 기준으로 보면, MAC 연산은 입력된 `pattern`과 `filter_matrix`를 받아
각 위치 `[i][j]`의 값을 곱하고, 그 결과를 `total`에 누적한 뒤 최종 점수를 반환하는 흐름으로 동작한다.

MAC 점수가 높다는 것은 같은 위치에서 값이 많이 일치했다는 의미이며,
즉 두 배열이 구조적으로 유사할수록 점수가 높아진다.

---

### 3.2 판정 방식

두 필터(Cross, X)의 점수를 비교하여 결과를 결정한다.

* Cross 점수 > X 점수 → Cross
* X 점수 > Cross 점수 → X
* 두 점수 차이가 매우 작은 경우 → UNDECIDED

부동소수점 오차를 고려하여 epsilon 기반 비교를 사용하였다.

```python
abs(score_cross - score_x) < 1e-9
```

모드 1은 사용자 입력 기반이므로 결과를 A/B/판정 불가로 출력하고,
모드 2는 표준 라벨 기반이므로 Cross/X/UNDECIDED로 출력한다.

---

### 3.3 부동소수점 오차와 epsilon 필요성

컴퓨터는 실수를 2진수로 표현하기 때문에 일부 값은 정확하게 저장되지 않는다.
이로 인해 계산 결과가 미세하게 다른 값으로 나타날 수 있다.

예시:

* 0.9000000000000000
* 0.8999999999999999

이러한 경우 단순 비교(`==`)를 하면 다른 값으로 판단된다.
따라서 두 값의 차이가 매우 작으면 같은 값으로 간주하기 위해 epsilon 기준을 사용하였다.

---

### 3.4 라벨 정규화

JSON 데이터에는 다양한 형태의 라벨이 존재할 수 있으므로, 내부적으로 표준 라벨로 변환하여 사용한다.

| 입력 값    | 변환 결과 |
| ------- | ----- |
| "+"     | Cross |
| "cross" | Cross |
| "x"     | X     |

라벨 정규화를 별도 함수로 분리한 이유는 입력 형식이 달라도 내부 로직을 동일하게 유지하기 위함이며,
새로운 라벨이 추가되더라도 유지보수가 용이하다.

---

### 3.5 JSON 키 파싱 및 필터 매칭

patterns의 키는 `size_5_1`, `size_13_2`와 같은 형식으로 구성된다.

이 문자열을 `_` 기준으로 나누어 두 번째 값을 추출하면 패턴 크기 N을 얻을 수 있다.

예시:

* size_5_1 → N = 5
* size_13_2 → N = 13

이 값을 이용하여 해당 크기의 필터를 선택하여 자동으로 매칭한다.

---

### 3.6 입력 검증

다음 검증을 수행한다.

* 행/열 개수 확인
* 숫자 형식 검증
* 필터와 패턴 크기 일치 여부 확인

잘못된 입력이 들어올 경우 오류 메시지를 출력하고 재입력을 유도하며, 프로그램은 종료되지 않는다.

---

### 3.7 성능 분석

MAC 연산의 실행 시간을 측정하기 위해 동일 연산을 10회 반복 실행한 후 평균 시간을 계산하였다.

시간 측정은 I/O를 제외하고 연산 함수 호출 구간만 대상으로 수행하였다.

출력 항목:

* 크기 (N×N)
* 평균 실행 시간 (ms)
* 연산 횟수 (N²)

모드 1에서는 두 필터(A, B)를 비교하므로 두 번의 MAC 연산을 묶어서 측정하였다.

## 4. 결과 리포트 (Result Report)

### 4.1 실행 결과 요약

* 총 테스트 수: 4
* 통과: 4
* 실패: 0

모든 테스트 케이스에서 expected 값과 실제 판정 결과가 일치하였다.

---

### 4.2 실패가 0인 이유

다음 요소들이 정확하게 구현되었기 때문이다.

* 라벨 정규화 적용
* epsilon 기반 비교 적용
* 입력 및 데이터 검증 수행
* 정확한 MAC 연산 구현

특히 부동소수점 오차를 고려한 epsilon 비교를 통해 미세한 값 차이로 인한 오판정을 방지하였다.

---

### 4.3 시간 복잡도 분석

MAC 연산은 N×N 행렬의 모든 원소를 한 번씩 처리하므로 총 연산 횟수는 N²이다.

```
O(N²)
```

이는 코드에서 이중 반복문(for i, for j)을 사용하여 각 원소를 한 번씩 순회하기 때문에 발생한다.

---

### 4.4 성능 분석 해석

행렬 크기가 증가할수록 연산 횟수가 N²으로 증가하므로 실행 시간도 함께 증가한다.
실제 측정 결과 또한 이론적인 시간 복잡도와 동일한 증가 경향을 보인다.

## 6. 확장 및 설계 고찰

새로운 라벨이 추가될 경우:

1. normalize_label 함수 수정
2. filters 정의 추가
3. 판정 로직 확장
4. 출력 로직 반영
5. 테스트 케이스 추가

패턴 크기가 매우 커질 경우(N=1000), 연산 횟수 증가로 인해 성능 저하가 발생할 수 있으며,
이 경우 1차원 배열 변환 등의 최적화를 고려할 수 있다.

CPU는 반복 연산을 순차적으로 처리하지만,
NPU는 병렬 처리에 최적화되어 있어 대규모 MAC 연산에서 더 효율적이다.
